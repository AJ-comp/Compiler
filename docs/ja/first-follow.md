# FIRST / FOLLOW

この章からLR構文解析の「心臓部」に入っていきます。\
最初の概念は **FIRST集合** と **FOLLOW集合** です。

正直に先に言っておきますね — 名前も聞き慣れないし、初めて見ると「これって一体何?」と思うかもしれません。\
**でも大丈夫です。みんな最初はここで一度つまずくものなんです。**\
コンパイラの授業で一番多くの人が迷う
ところがまさにここなんですよ。\
だから一度で腑に落ちなくても、まったくおかしなことではありません。\
本当にゆっくり、
手を取って進んでいきますね。\
読み終わるころには「あれ、たいしたことなかったな?」と思えるはずです。 🙂

---

> 📖 **始める前に:** この章は例題の文法を *読める* ことを前提にしています。`:` `|` `;` のような表記が
> 見慣れないなら、まず [文法の読み方](grammar-reading.md) を見てきてください — 5分もあれば十分です。

この章でずっと使う例題の文法です(参考用):

```
Expr   : Expr '+' Term | Term ;
Term   : Term '*' Factor | Factor ;
Factor : '(' Expr ')' | id ;
id     := "[a-zA-Z]+" ;
```

---

## ① なぜ必要なのか

パーサがトークンを左から一つずつ読み進めていくと、**絶え間なく決定を下さなければ** なりません。\
その中でも最も
重要な決定が次の二つです。

1. **「今、どの規則を始めてもよいのか?」** — 次のトークンを見て、どこに入るかを選ばなければならない
2. **「今読んでいた規則は終わったのか?」** — どこまでが一かたまりなのか、いつまとめるかを決めなければならない

言葉だけだと抽象的ですよね。\
例を挙げてみますね。\
`a + a * a` を読んでいて、**最初の `a` をちょうど見た** ところだとしましょう。\
パーサはこんな悩みに陥ります。

> 「この `a` 一つで一かたまり(`Term`)を終えたことにしようか? それとも後ろに `* 何か` がさらに付く、
> まだ終わっていないかたまりだろうか?」

どうやって判断するのでしょうか?\
意外と簡単です。\
**次のトークンをちらっと見ます。**

- 次が `*` → まだ終わっていない(掛け算がさらに付く)
- 次が `+` か入力の終わり → ここで終わり(まとめてよい)

つまりパーサは **「どんなトークンがこのかたまりの *次に* 来られるか」** を前もって知っていなければなりません。\
それがまさに
**FOLLOW** です。\
そして **「どんなトークンがこのかたまりを *始める* ことができるか」** が **FIRST** です。

> 💡 深く考えすぎないでください。一行でまとめるとこうです:
> **FIRST/FOLLOW = パーサが「始まり/終わり」を判断するために前もって作っておく *チートシート*。**
> これがあってこそ、次の章の [構文解析表](first-follow.md) を作ることができます。

## ② 何をするのか

> ここからは集合を実際に手で求めてみます。**計算が少し多く見えても気負わないでください。**
> パターンは単純ですし、ゆっくり一行ずつ一緒に埋めていきますから。

### FIRST — 「これはどんなトークンで *始まる* ことができるか」

**FIRST(X)** = X が作り出せるすべての文字列の **先頭に来られる終端記号(トークン)の集まり**。

一番やさしい `Factor` から行きましょう。

```
Factor : '(' Expr ')' | id ;
```

`Factor` は `(` で始まるか `id` で始まりますよね? だから:

```
FIRST(Factor) = { '(', id }
```

(`{ }` はただ「集まり、集合」という印です。\
波かっこの中に入っているのが候補たちです。)

次は `Term`:

```
Term : Term '*' Factor | Factor ;
```

`Term` は `Term` で始まるか(自分自身! — さっきのあの再帰ですね)`Factor` で始まります。\
自分自身で
始まるものを続けて展開していくと、結局先頭は `Factor` になります。\
だから:

```
FIRST(Term) = FIRST(Factor) = { '(', id }
```

同じ論理で `Expr` も:

```
FIRST(Expr) = { '(', id }
```

三つとも `(` か `id` で始まっていますね。\
納得できますよね?\
どんな式でも結局先頭は **名前(`id`)** か
**開きかっこ(`(`)** ですから。\
**ここまでついてこられたなら FIRST は終わり!**\
思ったよりたいしたことないでしょう?

### FOLLOW — 「これの *次に* どんなトークンが来られるか」

**FOLLOW(X)** = 正しい文のどこかで X の **すぐ後に現れることができる終端記号の集まり**。
ここには特別な記号 **`$`**(入力の終わりを意味する仮想のトークン)も入ることがあります。

`Expr` からですね。\
「`Expr` の次に何が来られるか?」を文法全体で探してみると:

- `Expr` は **開始記号** です(文全体が `Expr`) → だから `Expr` の次には **入力の終わり `$`** が来られる
- `Expr : Expr '+' Term` では → 最初の `Expr` の次には `+`
- `Factor : '(' Expr ')'` では → `Expr` の次には `)`

全部集めると:

```
FOLLOW(Expr) = { $, '+', ')' }
```

`Term` も同じように「Term の次に来るもの」を見渡してみると:

- `Expr` の末尾が `Term` である場合がありますよね(`Expr : ... Term`、`Expr : Term`) → そうすると **Expr の次に
  来られるものは Term の次にも来られます** → FOLLOW(Expr) がそのまま入ってくる
- `Term : Term '*' Factor` → 最初の `Term` の次には `*`

```
FOLLOW(Term) = FOLLOW(Expr) ∪ { '*' } = { $, '+', ')', '*' }
```

(`∪` は「合わせる(和集合)」の記号です。\
二つの集まりをただ合わせるという意味です。)

`Factor` も同じ式で:

```
FOLLOW(Factor) = { $, '+', ')', '*' }
```

### さあ、もう一度最初のあの悩みに戻りましょう

覚えていますよね?\
`a`(つまり `Factor` → `Term`)を読んで次のトークンを見たときのあの悩み。\
もう答えが見えてきます。

- 次が `*` → 「もっと進め」という信号(`Term '*' Factor` へ)
- 次が `+` または `$` → この二つは **FOLLOW(Term)** にある → 「Term は終わったから **まとめろ(reduce)**」

**まさにこうやって FOLLOW が「いつまとめるか」を決めるんです。**\
FIRST/FOLLOW がなぜ構文解析表の材料なのか、
今ちょっと感じが掴めてきましたか?(掴めなくても大丈夫です — 次の章で表を実際に見れば、ぐっと腑に落ちますから。)

### 特殊な状況が二つ(今は軽くだけ)

あまり深く入らなくていいです。\
「こういうものがあるんだな」くらいで。

- **ε(イプシロン、空文字列):** ある非終端記号が「何でもないもの」になれるときに使う印です。
  私たちの例題には出てこないので、今は気にしなくていいです。(あとで心化過程で別に扱います。)
- **`$`(終わりの印):** さっき見たように入力の終わりを表す仮想のトークンです。開始記号の FOLLOW には必ず入ります。
