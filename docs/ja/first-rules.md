# FIRST · 計算規則

> 🎓 **発展コース · 理論** です。\
> 前の [FIRST · 定義と導出](first-formula.md)では — *定義* をつかみ、*定義どおりに導出* もしてみましたね。\
> ところが **再帰では導出が無限に長くなってしまう壁** にぶつかりました。
>
> そこでこのページは — **導出を直接展開しなくても** 同じ FIRST を取り出せる **計算規則** です。\
> (再帰もおとなしく処理できます。)\
> その規則がコードでどう実装されているかは → **[FIRST · 実装](first-impl.md)**。

> 一度に全部を受け入れようとしないでください。\
> **やさしいものから一歩ずつ** 進みます。

## まず — 登場人物を全部並べてから始めます

FIRST を求める前に、この文法に *何があるのか* をひと目で見ておきましょう。\
文法の記号はちょうど **二種類** だけです。

- **終端記号(terminal)** — 入力に実際に現れるトークン
- **非終端記号(nonterminal)** — 規則の名前

私たちの例題文法で両者を分けて書くとこうなります。

```
   終端記号リスト   :   (    )    +    *    id        ← 5個
   非終端記号リスト :   Expr    Term    Factor        ← 3個
```

FIRST は *この記号ひとつひとつについて* 求めます。\
だから私たちがやることははっきりしています — **終端記号5個 + 非終端記号3個、合わせて8個の FIRST を埋めること。**\
そしてうれしい知らせ — **終端記号のほうはほぼタダ** です。そこから行きましょう。

## 終端記号の FIRST — すべて自分自身 (一発で終わり)

終端記号は *自分自身* で始まります。\
当然ですよね、`+` は `+` で始まるのですから。

```
   FIRST( '(' ) = { '(' }      FIRST( '+' ) = { '+' }      FIRST( id ) = { id }
   FIRST( ')' ) = { ')' }      FIRST( '*' ) = { '*' }
```

`FIRST(終端記号 a) = { a }`。\
終端記号リスト5個、これで **終わり。** 8個のうち5個をタダで埋めました。 🙂

## 非終端記号の FIRST — 大きな絵から

いよいよ本題、非終端記号の三つ(`Expr` `Term` `Factor`)です。\
その前に大きな絵を先につかんでおきましょう。

> 📎 ちょっと、用語をひとつ。`Factor : '(' Expr ')' | id` で `|` で分かれた **ひとつひとつ** を **生成規則
> (production)** と呼びます。\
> `Factor → id` のように *「非終端記号を作る規則の一行」* です。(詳しくは [Single](deep-single.md)。)\
> これからはこの言葉を使います。

非終端記号ひとつの FIRST はこう求めます — **その非終端記号のすべての生成規則それぞれの FIRST を求めて、全部
合わせる(和集合)** のです。

```
   FIRST(Factor) = FIRST(Factor の生成規則 1) ∪ FIRST(Factor の生成規則 2) ∪ …
```

だから本当に解くべき問いはひとつに絞られます — **生成規則ひとつの FIRST はどう求めるのか?**

答えは意外と単純です。\
その生成規則が **何で始まるか** にかかっていて、場合はちょうど **三つ** だけです。

1. **場合①** — 終端記号で始まるとき
2. **場合②** — 非終端記号で始まるとき
3. **場合③** — 先頭が消える(ε)可能性があるとき

やさしいものからひとつずつ見ていきましょう。

## 場合① — 生成規則が **終端記号で始まる** とき

いちばんやさしい場合です。実は **前で見たものをそのまま使う** だけです。

先頭が終端記号なら、その生成規則の FIRST はすなわち **その終端記号** です。\
すぐ上で *「終端記号の FIRST は自分自身」* と言いましたよね? **まさにそれと同じ話** です — 先頭の終端記号がそのまま答えです。\
(生成規則の後ろに記号がもっと付いていても関係ありません。先頭が終端記号なら、それが最初の終端記号なので、そこで終わるからです。)

たとえば `Factor` の二つの生成規則はそれぞれこうです。

```
   Factor → id            先頭が終端記号 id   →   FIRST = { id }
   Factor → '(' Expr ')'  先頭が終端記号 (    →   FIRST = { '(' }
```

`'(' Expr ')'` は後ろに `Expr ')'` がもっと付いていても、先頭の `(` ですぐ終わりますよね。

`Factor` はこの二つの生成規則だけなので、合わせればすぐ完成です。

```
   FIRST(Factor) = { '(' } ∪ { id } = { '(', id }
```

最初の非終端記号、終わり! 🙂

## 場合② — 生成規則が **非終端記号で始まる** とき

今度は先頭が終端記号ではなく、*また別の非終端記号* である場合です。

先頭の非終端記号を `Y` としてみましょう。\
この生成規則の FIRST は **`Y` の FIRST と同じです。**

**なぜそうなるのでしょう?**\
今回もまた定義に戻ります — FIRST は *「導出して出てくる先頭の終端記号」* でしたね。\
この生成規則を導出すると、先頭の位置を占めるのは `Y` です。\
すると最後まで導出して出てきた文字列の先頭の終端記号も、結局 **その `Y` が導出して出す先頭の終端記号** です。\
先頭を `Y` が握っているので、最初の終端記号も `Y` が決めることになるわけです。\
だから `Y` の FIRST をそのまま持ってくればよいのです。

たとえば `Expr` のある生成規則はこうです。

```
   Expr → Term     先頭が非終端記号 Term   →   FIRST(Term) をそのまま持ってくる
```

だからこの生成規則を通じて、`Expr` の FIRST に `FIRST(Term)` がそのまま入ってきます。\
式で書くとこう明快です — **`FIRST(Expr) = FIRST(Term)`**。\
(`Expr` の残りの生成規則 `Expr '+' Term` はすぐ見る *左再帰* なので新しく付け加えるものがなく、この等号がぴったり成り立ちます。)

### ところが — その非終端記号が *自分自身* なら? (左再帰)

ここで一度引っかかります。\
`Term : Term '*' Factor | Factor` の最初の生成規則を見ましょう。

```
   Term → Term '*' Factor     先頭がまた Term — 自分自身ですね?!
```

`FIRST(Term)` を求めようと見たら、先頭の非終端記号がまた `Term` です。\
つまり `FIRST(Term)` を求めるのに `FIRST(Term)` が必要な、**鶏が先か卵が先か** の状況ですね。\
このままでは一度には解けません。

そこでこう解きます — **空集合から始めて、もう増えなくなるまで一周ずつ繰り返す。**\
`FIRST(Term)` を実際に回してみましょう。(`FIRST(Factor) = { '(', id }` は場合①ですでに求めましたね。)

**開始 →** `FIRST(Term) = { }` (空集合)

**1周目 —** `Term` の二つの生成規則をたどります。

```
   ① Term → Factor          : FIRST(Factor) = { '(', id } を足す
                                          →  FIRST(Term) = { '(', id }
   ② Term → Term '*' Factor : 先頭 Term の「これまで」の値を足す
                                          →  現在の値 { '(', id }、すでにある → 変化なし
```

→ 空集合から `{ '(', id }` へ **増えました。**\
増えたのでもう一周回らなければなりませんね。

**2周目 —** ふたたび二つの生成規則をたどります。

```
   ① Term → Factor          : { '(', id }                  →  変化なし
   ② Term → Term '*' Factor : Term の現在の値 { '(', id }    →  変化なし
```

→ この周では **何も増えませんでした。→ 停止!**

```
   FIRST(Term) = { '(', id }
```

カギは生成規則②です。\
自分自身を指してはいますが、「先頭の `Term` の *これまで* の値」を引っぱってくるだけなので、その値がほかの生成規則
①(`Factor`)で **先に埋まってしまえば** もう付け加えるものがなくなります。\
だから無限に展開しなくても、二周で答えが固まります。

> 💡 前のページ([定義と導出](first-formula.md))で導出が *無限に長くなっていた* あの再帰の壁 — **その壁を
> 越えるのがまさにこの「繰り返し」** です。\
> 最後まで展開する代わりに、集合を少しずつ大きくして変わらなくなったら止めるからです。

`Expr`(`Expr : Expr '+' Term | Term`)も同じ形なので、同じやり方で二周すれば `{ '(', id }` に
固まります。

## 場合③ — 先頭の非終端記号が **消える可能性があるとき** (ε)

最後の場合です。\
場合②で *「先頭の非終端記号 `Y` の FIRST を持ってくる」* と言いましたね?\
ところがもしその `Y` が **空(ε)で消えてしまう可能性も** あるなら、もうひとつ気をつけることがあります。\
(ある非終端記号が空文字列まで導出できることを *nullable* と呼びます。)

**なぜ次の記号まで見なければならないのでしょう?**\
やはり定義です — FIRST は *「導出して出てくる先頭の終端記号」* でしたね。\
ところが先頭の `Y` が ε で消えると、導出結果の先頭を占めるのは `Y` ではなく **すぐ次の記号** です。\
するとその次の記号が導出する最初の終端記号も先頭に来うるわけです。\
だから `Y` の FIRST に **その次の記号の FIRST まで** 足さないと正しくありません。\
この *「先頭が消える可能性があれば次の記号へ進みながら合わせる」* 規則を **⊕(リングサム)** と呼びます。

```
   A ⊕ B =  A              (A が消えられないなら → そこで終わり)
            (A-ε) ∪ B      (A が消えられるなら → ε を取り除き、B も足す)
```

> 幸い私たちの例題文法には *消える(nullable)非終端記号* がひとつもありません。\
> だから場合③は実際には起きず、⊕ もほぼ先頭の記号ですぐ終わります。\
> 今は **「こんな安全装置があるんだな」** とだけ知っておけば十分です。⊕ の真価は ε のある文法で現れます。

## まとめ

生成規則ひとつの FIRST は、それが **何で始まるか** で分かれました。

1. **終端記号で始まる** → その終端記号ひとつ。(場合①)
2. **非終端記号で始まる** → その非終端記号の FIRST をそのまま。(場合②、自分自身なら *繰り返し* で)
3. **先頭が消える可能性があれば(ε)** → 次の記号の FIRST まで ⊕。(場合③)

そして非終端記号ひとつの FIRST は、**その非終端記号のすべての生成規則の FIRST を和集合** したものです。

この規則を私たちの文法に回すと、三つとも同じ答えになります。

```
   FIRST(Factor) = FIRST(Term) = FIRST(Expr) = { '(', id }
```

[定義と導出のページ](first-formula.md)で手で求めた答えと正確に同じです。 ✓

## もう一歩 — 三つの場合は結局 ⊕(リングサム)ひとつです

いま三つに分けて見ましたが、実はこの三つは **ひとつの式** に合わさります。

生成規則は結局 *記号たちの列* ですよね(`Term '*' Factor` のように)。\
その FIRST は — **構成記号たちの FIRST を順番に ⊕(リングサム)したもの**、それがすべてです。

```
   FIRST(X₁ X₂ … Xₙ) = FIRST(X₁) ⊕ FIRST(X₂) ⊕ … ⊕ FIRST(Xₙ)
```

すると前の三つの場合は — この **⊕ が *どこで止まるか*** の違いにすぎません。

- **場合①** : 最初の記号が終端記号です。終端記号は ε にならないので、⊕ が **最初のマスで止まって** `{ その終端記号 }` になります。
- **場合②** : 最初の記号が(ε にならない)非終端記号です。⊕ がその **FIRST で止まって** `FIRST(その非終端記号)` になります。
- **場合③** : 最初の記号が ε になりえます。だから ⊕ が **次のマスへ進みながら** 合わせ続けます。

例として `Term '*' Factor` をそのまま ⊕ してみるとこうです。

```
   FIRST(Term '*' Factor) = FIRST(Term) ⊕ FIRST('*') ⊕ FIRST(Factor)
                          = FIRST(Term)              ← Term は ε にならないので最初のマスで止まる
```

コードもまさにこれです — 生成規則([Concat](deep-concat.md))の記号たちを順に `RingSum(⊕)` していって、
これ以上見る ε がなくなれば止まります。

```csharp
// FirstFollowAnalyzer [First].cs
public TerminalSet FirstSet(NonTerminalConcat singleNT, ...)
{
    TerminalSet result = new TerminalSet();

    foreach (var symbol in singleNT)                        // 生成規則の記号を順番に
    {
        result = result.RingSum(FirstSet(symbol, seenNT));  // ⊕ 一マス
        if (!result.IsNullAble) break;                      // 見るべき ε がなくなれば止まる
    }

    return result;
}
```

> 💡 **一行要約** — 生成規則の FIRST は *その記号たちの FIRST を順番に ⊕ したもの* で、三つの場合は
> その ⊕ が止まる位置が違うだけです。
