# 構文解析表 · 衝突とは？

前のページで私たちは Expr 文法を使って**構文解析表**を完成させました。その表をちょっとだけ思い出してみましょう。\
一マス一マスに行動が**ちょうど一つずつ**入っていましたね — `s5`、`r2`、`goto 3`、`acc` … だからパーサーはその都度*迷わずに*マスを一つ見つけ、そのとおりに従うだけでよかったのです。

ところが作り方ページの最後で、私はそっと一つの問いを置いておきました。

> **「もし一つのマスに行動が*二つ*入ったらどうしましょう？」**

このページがまさにその話です。ゆっくり進んでいきましょう。

---

## まず — 表はなぜ「詰まることなく」回ったのか

構文解析表のすることを一行にまとめると、これでした。

> **いまの状態 + 次に来る文字 → どんな行動をするか**、それを一マス一マスに書いておいたもの。

パーサーはその都度*いまの状態*と*次の文字*で表からマスを一つ選び、そこに書かれたとおりにだけ動きます。\
マスごとに書かれた答えが**一つ**だから — パーサーは一度も悩みません。このように*悩みなくスイスイ回る*性質を**決定的（deterministic）**と呼びます。

> 🔖 **決定的（deterministic）** — その都度すべき行動が*ちょうど一つに*定まっていて、パーサーが迷うことがないこと。

さて、この**「マスごとに一つ」**という約束が破れると、何が起きるのでしょうか？

---

## 衝突 — 一つのマスに行動が二つ

ある文法では — 表を埋めていくと**一つのマスに行動が二つ**入ってしまいます。\
そのマスに着いたパーサーはその瞬間ためらいます。*「えっ… 二つのうちどっちをしろってこと？」*

このように**一つのマスに入る行動が二つ以上で、パーサーが何をすべきか決められない状況**を**衝突（conflict）**と呼びます。

> 🔖 **衝突（conflict）** — 構文解析表の一つのマスに行動が二つ以上入って、パーサーが決定できなくなったこと。

私たちの Expr 文法は運よく — 実は*よく設計されていたので* — 衝突が一つもありませんでした。だから表があんなにきれいだったのです。\
でも世の中の多くの文法はそうではありません。**衝突がどうやって生まれるのか**、その二つの種類を一つずつ見てみましょう。

---

## 種類① — shift / reduce 衝突

衝突を*わざと*見せるために、私たちの Expr ではなく、とても小さな文法を一つ持ってきます。

<pre class="lrbox">
<span class="nt">S</span> → <span class="nt">S</span> <span class="setm">'+'</span> <span class="nt">S</span>
<span class="nt">S</span> → <span class="setm">a</span>
</pre>

「S は *S 足す S*、または単に *a*」 — 足し算を表すちっぽけな文法です。

この文法で[状態](lr-state.md)たちを作っていくと、ある一つの状態の中にこの**二つの項目が一緒に**入ります。（項目・状態がうろ覚えなら [LR項目](lr-item.md)・[状態](lr-state.md)の章を思い出してください。）

<pre class="lrbox">
<span class="nt">S</span> → <span class="nt">S</span> <span class="setm">'+'</span> <span class="nt">S</span> <span class="lrdot">•</span>      <span style="opacity:.65">← ドットが末尾。<b>完了</b>項目 → '+' を見たら reduce（S+S を S にまとめる）</span>
<span class="nt">S</span> → <span class="nt">S</span> <span class="lrdot">•</span> <span class="setm">'+'</span> <span class="nt">S</span>      <span style="opacity:.65">← ドットの後ろが '+'。<b>進行中</b>項目 → '+' を見たら shift（'+' を読んでさらに進む）</span>
</pre>

同じ状態、同じ次の文字 `+` なのに —\
**上の項目は「reduce しろ」**と、**下の項目は「shift しろ」**と、互いに*違う行動*を叫びます。

ですからこの状態の `+` のマスには行動が**二つ**入ってしまいます。

| 状態 | … | `+` | … |
|:--|:--:|:--:|:--:|
| （この状態） | | **shift / reduce** ⚠️ | |

片方は **shift**、片方は **reduce** なのでぶつかるこの衝突を — そのまま **shift/reduce 衝突**と呼びます。

> 💡 **なぜ生じたのか** — `a + a + a` を思い浮かべてみてください。真ん中まで読んだパーサーは、*左の `a + a` を先に一塊にまとめるか*（reduce）、*それともひとまず `+` をさらに受け取って後で決めるか*（shift）を決めなければなりません。ところがこの文法は `(a+a)+a` も、`a+(a+a)` も**どちらも正しい**と言います。答えが二つだからパーサーが選べないのです。このように*一つの文を二通りに解釈できる*文法を**あいまいな（ambiguous）文法**と呼びます。（ただし*衝突が即あいまいさというわけではありません* — あいまいでないのに衝突が見える場合があり、そのうちのかなりが、表をもっと精密に埋めれば消える**見かけの衝突**です。まさに次の話ですね。）

---

## 種類② — reduce / reduce 衝突

今度はまた別の小さな文法。

<pre class="lrbox">
<span class="nt">S</span> → <span class="nt">A</span>
<span class="nt">S</span> → <span class="nt">B</span>
<span class="nt">A</span> → <span class="setm">a</span>
<span class="nt">B</span> → <span class="setm">a</span>
</pre>

「`a` 一つが *A* にもなれるし *B* にもなれる」文法です。

表を作っていくと、`a` を読んだ直後の状態にこの二つの項目が一緒に入ります。

<pre class="lrbox">
<span class="nt">A</span> → <span class="setm">a</span> <span class="lrdot">•</span>      <span style="opacity:.65">← 完了。reduce <b>A→a</b></span>
<span class="nt">B</span> → <span class="setm">a</span> <span class="lrdot">•</span>      <span style="opacity:.65">← 完了。reduce <b>B→a</b></span>
</pre>

今度は両方とも **reduce** です。ところが — **どの規則で**まとめろと言うのでしょう？`A→a` で？`B→a` で？\
どちらも「まとめろ」なのに*まとめる対象が違うので*ぶつかります。これを **reduce/reduce 衝突**と呼びます。

二つの種類を並べてみると:

| 衝突の種類 | 一つのマスでぶつかるもの | パーサーの悩み |
|:--|:--|:--|
| **shift / reduce** | shift 一つ + reduce 一つ | 「もっと読むか、いままとめるか？」 |
| **reduce / reduce** | reduce 二つ（規則が違う） | 「どの規則でまとめる？」 |

---

## [状態]の章で植えておいた種

覚えていますか？[状態](lr-state.md)の章で — 一つの状態の中には**完了項目**（ドットが末尾にあって*まとめようとする*）と**進行中項目**（ドットの後ろに記号が残っていて*読もうとする*）が**混ざっていることがある**と言って、それを🌱**衝突の種**と呼んでおきました。

いま、その種が育って実になったのです。

- 一つの状態で**完了項目と進行中項目**が*同じ文字*をめぐってぶつかれば → **shift/reduce 衝突**
- 一つの状態で**完了項目が二つ**ぶつかれば → **reduce/reduce 衝突**

状態を学ぶときに「これが後で衝突になります」と先送りしておいたあの話を、いまようやく全部返した格好です。

---

## それで — 衝突が出るとどうなるのか

文法に衝突があると、その表は**「マスごとに答えが一つ」**という約束が破れます。\
= その文法は*そのままでは*決定的に構文解析できません。パーサーがそのマスで止まってしまうからです。

抜け出す道は二つです。

1. **文法を直す** — あいまいさを取り除いて、衝突がそもそも出ないように書き直す。\
   （私たちの Expr 文法がまさにそうやって*よく整えられた*文法です。だから表がきれいだったのですね。）
2. **表をもっと賢く埋める** — 驚くことに、衝突のように*見える*もののうち一部は**本当の衝突ではなく**、表を*精密でなく*埋めたせいで生じた**見かけの衝突**です。もっと精密に埋めるとするりと消えます。

その**「もっと精密に埋める」**へ向かう旅が — まさに次のページ（SLR）から始まります。\
作り方ページで reduce のマスを **FOLLOW** で決めたあの方式（名前は **SLR** でしたね？）が*なぜ見かけの衝突*を生むのかから見て — そこから **lookahead → CLR → LALR** と精度のはしごを一段ずつ上りながら、その見かけの衝突を*どうやってなくすか*を見ます。
