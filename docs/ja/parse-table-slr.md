# 構文解析表 · SLR と見かけの衝突

前の章で衝突に出会いました。そして最後にこんなことをそっと漏らしておきましたね —

> 衝突には*本物*もあるけれど、表を**精密でなく埋めて生じた偽物**もある。

この章でその**「精度」**の話を始めます。ところが不思議なことに — これがすべて**たった一つの問い**から分かれて出てきます。

---

## すべてはこの一つの問いから

衝突はほとんどいつも**完了項目**のせいで生じます。状態の中に `A → α •` のようにドットが末尾まで行った項目があると*「もう reduce しろ」*という意味ですね。ところが —

> **「`A → α •` に出会ったとき、*どんな次の文字*で reduce を書くべきか？」**

まさにこの問いです。\
この答えを*どれだけ精密に*するかが — 衝突が出るか出ないかを分け、これから出てくる **SLR · CLR · LALR** という名前たちが、実は全部*この一つの問いに対する互いに違う答え*なのです。

> 🔖 **lookahead（先読み）** — パーサーがいま決定を下そうとして*ちらっと見る次の文字一つ*。（[閉包](closure-def.md)の章で一度かすめたあの言葉です。）

さあ、精度を一段ずつ上げていきましょう。

---

## ① 最も粗い答え — 「どんな文字でも」（LR(0)）

いちばん単純な答え。\
`A → α •` なら、*次の文字が何であっても* reduce。

これが私たちが[正準集合](canonical-set.md)まで作った **LR(0) 状態**そのままの答えです — その状態たちには*次の文字についての情報がそもそもないから*です。

問題はわかりきっています。reduce をあまりに無造作に書くので、同じ状態に shift 項目が少しでもあると*たちまち*ぶつかります。たいていの実際の文法は、この段階で衝突だらけになります。\
→ もっと精密でなければなりません。

---

## ② 一段上 — 「FOLLOW でだけ」（SLR）

賢い第一歩。\
`A → α •` をどこでもではなく — **A の後ろに来られる文字でだけ** reduce するのです。その*「A の後ろに来られる文字たち」*がまさに [FOLLOW(A)](follow-formula.md) でしたね。

これが[作り方](parse-table-build.md)ページで私たちが実際に使った方式で、名前が **SLR** です。

> 🔖 **SLR (Simple LR)** — LR(0) 状態はそのままにして、reduce は **FOLLOW(A) に入った文字でだけ**書く方式。（"Simple" = 最も手軽な一段の改善という意味。）

**ただ通り過ぎずに、目で一度見てみましょう。** 私たちの Expr 文法で状態を作っていくと — *Term を読んだ直後*にこんな状態が一つ出てきます。

<pre class="lrbox">
<span class="nt">Expr</span> → <span class="nt">Term</span> <span class="lrdot">•</span>              <span style="opacity:.65">← 完了。Expr→Term にまとめようか？</span>
<span class="nt">Term</span> → <span class="nt">Term</span> <span class="lrdot">•</span> <span class="setm">'*'</span> <span class="nt">Factor</span>    <span style="opacity:.65">← 進行中。'*' を見たら shift</span>
</pre>

この状態で**次の文字が `*`** のとき — 決めることはたった一つです。\
*上の `Expr → Term •` をいままとめるか（reduce）、やめるか？*

二つの方式がこの問いに**違うふうに**答えます。

**LR(0) の答え — 「無条件にまとめろ。」**\
完了項目（`Expr → Term •`）は*次の文字をまったく見ません。*どんな文字が来てもただ「まとめろ」です。\
だから `*` でも「Expr→Term にまとめろ」と書いてしまいますね。ところがすぐ下の項目（`Term → Term • '*' Factor`）は `*` で*shift*しろと言います。\
→ 一つのマスに「まとめろ」と「もっと読め」が一緒に → **衝突！** ⚠️

**SLR の答え — 「`Expr` の後ろに来られる文字のときだけまとめろ。」**

ところが — *なぜ*よりによって「来られる文字のときだけ」なのでしょう？ここに論理があります。

`Expr → Term •` をまとめるということは、いま作った Term を **`Expr` 一塊にまとめてスタックに載せる**という意味です。そうしたらパーサーは*その `Expr` のすぐ後ろに*入力の次の文字を続けて見ることになりますね。

だからまとめてもよいためには — **その次の文字が、`Expr` の後ろに*本当に来られる*文字でなければ**なりません。\
もし `Expr` の後ろに*絶対に来られない*文字なら？それをまとめた瞬間、**「`Expr` の次にその文字」**という — *文法のどこにもない形*を作ってしまうのです。それはつまり*「ここでまとめると間違う」*という意味ですね。

そして**「`Expr` の後ろに来られる文字」** — これがまさに [FOLLOW](follow-formula.md) の**定義**でした。

> 🔖 **`FOLLOW(Expr)`** = `Expr` のすぐ後ろに来られる終端記号の集合。*（定義そのまま。）*

だから SLR のルールは*定義からまっすぐ従って出てきます* — **次の文字が `FOLLOW(Expr)` の中にあるときだけまとめる。**

**さて、では `*` は？** 定義どおり `FOLLOW(Expr)` を*直接求めて*みましょう。\
FOLLOW を求める方法は — 文法で**`Expr` が登場する場所ごとに、その*すぐ後ろ*に何が来るか**を集めることでしたね。`Expr` が右辺に出てくる規則をもれなく探してみると:

| `Expr` が出てくる規則 | `Expr` のすぐ後ろに来るもの |
|:--|:--:|
| `Expr → Expr '+' Term` | `+` |
| `Factor → '(' Expr ')'` | `)` |
| `Accept → Expr`（増補規則） | 入力の終わり `$` |

これが*すべて*です。だから — <code>FOLLOW(Expr) = <span class="setb">{</span><span class="setm"> + ) $ </span><span class="setb">}</span></code> です。\
（<span class="setb">{ }</span> は*集合*を表す記号にすぎず、中に入った `+` · `)` · `$` が実際の要素です。）

ここで決定的な一つが見えます — **どの規則でも `Expr` の後ろに `*` が来ません。**\
そうならざるをえません。`*` は文法で唯一 `Term → Term '*' Factor` **の一か所だけに**出てくるからです。つまり `*` はいつも*`Term` のすぐ後ろ*の場所で、*`Expr` の後ろ*の場所では**ありません。**

これで論理が閉じます:

> `*` は `FOLLOW(Expr)` にない（= `Expr` の後ろには `*` が来られない）。\
> だから `*` で `Expr → Term •` をまとめると*「`Expr` の次に `*`」*という**文法にない形**になる。\
> したがって SLR は `*` では**まとめない。**

reduce を書かないのでそのマスには*shift 一つだけ*が残り → **衝突が消えます。** ✅

| 方式 | `*` ですること |
|:--|:--|
| **LR(0)** | 次の文字を見ずに*無条件に*まとめる + shift → ⚠️ 衝突 |
| **SLR** | `*` は Expr の後ろに来られないので*まとめない* → shift だけ → ✅ きれい |

**一行の直観:** `a * …` を読んでいる途中なら — 当然*掛け算をもっと受け取るべき*で、そこでまるごと `Expr` にまとめてはいけませんよね？SLR は*「`*` は `Expr` の後ろに来られない文字」*という事実一つで、まさにその**「ここではまとめるな」**を正確に見抜いたのです。

このように — *「この文字が Expr の後ろに来られるか」*だけを一度問うたおかげで、私たちの Expr 文法は SLR で衝突が**0**になります。[作り方](parse-table-build.md)の表があんなにきれいだったのは、まさにこのおかげです。\
SLR だけでもかなり多くの文法がうまく解けます。ところが —

---

## ③ SLR の弱点 — 見かけの衝突

`FOLLOW(A)` が何だったか、もう一度思い出してみましょう。\
それは**文法全体**を探って、A が*どこに現れようと*その後ろに来られる文字を*ことごとく*集めた集合です。**いまこの状態とは関係なく。**

まさにここに落とし穴があります。

いまパーサーが立っている**この状態**は、文法の中の*特定の一つの場所*です。\
その場所で A の後ろに*実際に*来られる文字は — FOLLOW(A) **全体ではなく、そのうちの一部**であることが多いのです。

ところが SLR は*全体*の FOLLOW(A) を持ってきて reduce を書きます。そうしているうちに —\
**この状態では絶対に来ない文字**にまで reduce を書いてしまい、それが隣にあった shift とぶつかります。\
文法が*本当に*あいまいなわけではないのに、ただ*広く書きすぎて*生じた衝突 — これを**見かけの衝突（spurious conflict）**と呼びます。

> 🔖 **見かけの衝突（spurious conflict）** — 文法が実際にあいまいなのではなく、reduce を*必要以上に広い文字*に書いて生じた*偽物*の衝突。

少し抽象的ですね。**小さな例題で直接見てみましょう。**

<pre class="lrbox">
<span class="nt">S</span> → <span class="setm">a</span> <span class="nt">A</span> <span class="setm">c</span>
<span class="nt">S</span> → <span class="setm">a</span> <span class="nt">B</span> <span class="setm">d</span>
<span class="nt">S</span> → <span class="setm">e</span> <span class="nt">A</span> <span class="setm">d</span>
<span class="nt">A</span> → <span class="setm">b</span>
<span class="nt">B</span> → <span class="setm">b</span>
</pre>

`A` も `b` に、`B` も `b` に広がる — わざと少し紛らわしく作った文法です。（それでも*あいまいではありません*。前後の文字で十分に分かれますから。）

まず `FOLLOW(A)` を求めます。`A` は**二か所**出てきます — `a A c`（後ろに `c`）と `e A d`（後ろに `d`）。だから <code>FOLLOW(A) = <span class="setb">{</span><span class="setm"> c d </span><span class="setb">}</span></code> です。（`B` は `a B d` の一か所だけなので <code>FOLLOW(B) = <span class="setb">{</span><span class="setm"> d </span><span class="setb">}</span></code>。）

では、パーサーが **`a b` まで読んだ状態**を見ましょう。この状態には完了項目が二つ入っています。

<pre class="lrbox">
<span class="nt">A</span> → <span class="setm">b</span> <span class="lrdot">•</span>      <span style="opacity:.65">← 完了</span>
<span class="nt">B</span> → <span class="setm">b</span> <span class="lrdot">•</span>      <span style="opacity:.65">← 完了</span>
</pre>

SLR で reduce のマスを埋めると:

- `A → b •` は <code>FOLLOW(A) = <span class="setb">{</span><span class="setm"> c d </span><span class="setb">}</span></code> で reduce
- `B → b •` は <code>FOLLOW(B) = <span class="setb">{</span><span class="setm"> d </span><span class="setb">}</span></code> で reduce
- → **`d` で両方とも reduce！** reduce/reduce 衝突 ⚠️

**ところがこれは本当の衝突でしょうか？** `a b` の次を考えてみましょう。\
私たちは `a` で始めたので — 行ける道は `a A c` か `a B d`、二つだけです。

- 次が `c` なら → `a A c` の道 → `A` にまとめるのが正しく
- 次が `d` なら → `a B d` の道 → `B` にまとめるのが正しいです

つまり **`d` では必ず `B` です。ここで `A` にまとめることは*絶対に*ありません。**

では SLR はなぜ `d` のマスに `A → b` reduce を書いたのでしょう？\
`A → b` の reduce を <code>FOLLOW(A) = <span class="setb">{</span><span class="setm"> c d </span><span class="setb">}</span></code> *全体*で書いたからです。そしてその `d` は — *いまこの `a` の道*ではなく、**はるか遠くの `e A d` 規則**から流れ込んだものです。`e A d` では `A` の後ろに `d` が来ますが、*いま私たちが立っている `a` の道*では `A` の後ろには `c` だけが来るのですから。

**これがまさに見かけの衝突です。** 文法があいまいなのではなく — *まったく別の場所（`e A d`）の事情*まで `FOLLOW` にひとまとめにして詰め込んだせいで、*いまの場所にはなくてもよい reduce* が書かれたのですね。

> これは[衝突とは？](parse-table-conflict.md)の章の*本物の* reduce/reduce とは違います。あちらは文法が*本当に*あいまいでしたが（同じ `a` を A とも B とも見られましたね）、こちらはあいまいでないのに SLR が*偽物*を見たのです。 — まさに**はしごの上の段（CLR）が正確に直す地点**です。（CLR の章で*まさにこの文法*をもう一度広げてお見せします。）

たとえでもう一度 —\
`FOLLOW(A)` は*「A という人が生きてきて一度でも出会ったすべての人の名簿」*のようなものです。\
ところがいまこの状態は*「今日、この部屋」*です。\
今日この部屋に来る人はその名簿の一部だけなのに — *名簿全体*を見て「この人も来るかもしれないから席を空けておけ」とやるので、来もしない人の席のせいで*見当違いにぶつかる*のです。

---

## それで — 次へ

この見かけの衝突は — FOLLOW *全体*ではなく**「まさにこの状態に本当に到達する文字」**にだけ reduce すれば消えます。その*「この場所にぴったり合う文字」*がまさに次の章の **lookahead** で — それを状態に最初から埋め込むのが **CLR**、まとめて縮めたのが **LALR** です。一歩ずつ上っていきます。

その前に — いま学んだ SLR が*エンジンのコード*ではどんな姿をしているか、たった一ページだけ見ていきましょう。（言葉で説明した定義がコードにそのまま埋まっているので、一度見るとさらにくっきりします。）
